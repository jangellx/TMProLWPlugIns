Integration Basics

To add SpaceWare support to an application, you add the appropriate library and header files and use at least five basic functions.
 

Library Files for Applications

The SpaceWare SDK for Windows contains two libraries:

SpaceWare Input Library (siapp.lib)
SpaceWare Math Library (spwmath.lib)

These libraries are both contained in siappmt.lib (for multithreaded applications) or siappst.lib (for single-threaded applications); thus a user only has to link with one library to add Spaceball support using the shipped libraries.

The SpaceWare Input Library is designed to enable application developers to quickly get Spaceball data into an application and to insulate them from the details of getting this data.

The SpaceWare Math Library contains utility functions for converting the Spaceball data into matrices and for manipulating matrices and vectors.

 

Five Basic Functions

There are five functions you need to use to establish communication with the Spaceball and to check to see in an event is a Spaceball event. They are:

  1. SiInitialize tells the SpaceWare library to set up certain internal states. It should be the first SpaceWare function you call. It also needs to be called if an application has stopped receiving input (as happens when you call SiTerminate) and you want it to start again.
  2. SiOpenWinInit is a Windows-specific call that tells the SpaceWare library which window should receive data. It takes two parameters:
      - a pointer to a SiOpenData Structure for platform-specific data returned by SiOpenWinInit
      - an HWND to which you send data. This window ID handle must be valid.
  3. SiOpen makes a connection to the SpaceWare driver for the application to receive the data. This function takes several parameters:
      - a character string specifying the application name
      - a device ID, the constant SI_ANY_DEVICE
      - a device type mask, the constant SI_NO_MASK
      - the event retrieval method, SI_EVENT
      - a pointer to SiOpenData from SiOpenWinInit

    This function returns a SiHdl reference to the device opened with SiOpen if it succeeds, or NULL if there was some type of connection error.

    Once SiOpen has been called and a connection established, the Spaceball driver starts sending Spaceball messages to the specified window.

  4. SiGetEventWinInit initializes Windows specific data structures. This function call needs:
      - a pointer to SiGetEvent Data Structure that is filled in by this function call
      - Windows message ID
      - WPARAM and LPARAM
  5. SiGetEvent returns SI_IS_EVENT if the event is a Spaceball event. SiGetEvent has other return values (SI_NOT_EVENT, SI_BAD_HANDLE, SI_SKIP_EVENT) that should be checked for more robust error handling.

    SiGetEvent takes four arguments:

      - the SiHdl device handle from SiOpen
      - the event processing flag, which should be NULL
      - a pointer to a SiGetEvent Data Structure for platform-specific data
      - a pointer to a SiSpwEvent structure, which is filled in if the function returns SI_IS_EVENT

     

Spaceball Events

There are three types of Spaceball events (SiSpwEvent):

  • SI_MOTION_EVENT
  • SI_ZERO_EVENT
  • SI_BUTTON_EVENT

A SI_MOTION_EVENT is returned if the user is moving the PowerSensor on the Spaceball.

A single SI_ZERO_EVENT is returned when the user stops moving the PowerSensor. This can be used in CAD systems, for example, to signal a shaded redraw. The values for the motion data are contained in the mData field of the SiSpwEvent Structure, in the following format:

siSpwEvent.u.spwData.mData[0] = Translation along X axis
siSpwEvent.u.spwData.mData[1] = Translation along Y axis
siSpwEvent.u.spwData.mData[2] = Translation along Z axis
siSpwEvent.u.spwData.mData[3] = Rotation about X axis
siSpwEvent.u.spwData.mData[4] = Rotation about Y axis
siSpwEvent.u.spwData.mData[5] = Rotation about Z axis
siSpwEvent.u.spwData.period = Time in milliseconds since the last event

See the Data Supplied and How Data Should be Used sections for more information on using the motion data.

A SI_BUTTON_EVENT can be either a button press or a button release. To determine if the event was a press or a release, call SiButtonPressed or SiButtonReleased with the returned SiSpwEvent as the only parameter. These functions each return an integer representing the corresponding Spaceball button, or SI_NO_BUTTON if the event was not of the correct type. For more intensive button processing you can examine the SiButtonData Structure, which provides the current and last states of each button on the Spaceball.

 

Spaceball Button Returns

All Spaceball devices have buttons. Each button returns a numerical value when pressed (or, in the case of the 3003, toggles the driver menu Open/Closed). The older Spaceball 2003 series has a button on the back of the PowerSensor (the 'Pick' button) which does not exist on the newer FLX series. The Spaceball 3003 has two buttons, Left and Right. Future versions of the Spaceball device may have up to 32 buttons, so it is best to structure any internal switch statements with the appropriate default cases. 


The three constants, SI_RESET_BUTTON, SI_PICK_BUTTON, and SI_DIALOG_BUTTON give the application some hints at how a particular button should be used.

The Pick button (SI_PICK_BUTTON) is typically used to reset or recenter the view.

The Reset button (SI_RESET_BUTTON) is normally used to rezero the Spaceball.

The SI_DIALOG_BUTTON is normally used to bring up the driver menu or an application specific GUI related to the Spaceball.

 

Spaceball Driver Menu

The SiSetUiMode function controls the behavior of the GUI (see the description of that function for a full list of the arguments). To have your application access the driver, execute:

SiSetUiMode(hdl, SI_UI_ALL_CONTROLS)

For an application that has its own GUI and does not use the Spaceball GUI, execute:

SiSetUiMode(hdl, SI_UI_NO_CONTROLS)

When you want the application to stop receiving Spaceball data, call the functions:

SiClose (hdl);
SiTerminate ();

After you call these functions, it is necessary to call SiInitialize and SiOpen to start receiving Spaceball data again.

 

Spaceball Utility Functions

Input Library

SiBeep enables the caller to cause the Spaceball to beep.

SiRezero lets the caller cause the Spaceball to set its current position as the rest position (stopping drift or setting up constant motion).

SiGetDeviceInfo returns information about the Spaceball device, such as type and number of buttons.

SpwErrorString can be called (after any function that returns a SI_ constant) to return a character string with more information about the error condition or return value. Additionally, the global variable SpwErrorVal can be used in conjunction with SpwErrorString to get more information from functions like SiOpen which do not return an error value.

 

Math Library

While a majority of the SpaceWare Math Library functions are useful utilities, two deserve special attention:

SPW_ArbitraryAxisToMatrix converts an arbitrary rotation vector into a matrix. It is highly recommended that an application use this function to apply the Spaceball rotation values. Treating the Spaceball rotation vector as three angles can lead to wobble during rotation (see Why ArbitraryAxisToMatrix for more details).

SPW_HighValuePassFilter filters out all but the largest value in an array. This is useful in setting up a single axis filter for the Spaceball data, giving a user fine control of the motion applied.

 

Code Samples

The following examples show how to support the Spaceball using different Windows programming approaches.

 

Win32

/*
 * SpaceWare include file
 */
#include “si.h”

int WINAPI WinMain ( HINSTANCE hInstance, HINSTANCE hPrevInstance,
                     LPSTR lpszCmdLine, int nCmdShow )
{
    SiOpenData oData;      /* Platform-specific device data */
    SiGetEventData eData;  /* Platform-specific event data */
    SiHdl hdl;             /* Spaceball device handle */
    SiSpwEvent siEvent;    /* SpaceWare data event */

/* Application functions as needed */

/*
 * Attempt to connect with the Spaceball driver.
 */

  SiOpenWinInit (&oData, hWnd);
  if ( (hdl = SiOpen ("app name", SI_ANY_DEVICE, SI_NO_MASK,
                       SI_EVENT, &oData)) == SI_NO_HANDLE);
    {
    /* Get and display initialization error */
    MessageBox (hWnd, "Could not open Spaceball", NULL,
                MB_ICONSTOP | MB_OK);
    }

  /*
   * Main messaging loop.
   */
  while (GetMessage (&msg, NULL, 0, 0))
    {
    SiGetEventWinInit (&eData, msg.message, msg.wParam,
                       msg.lParam);
    if (SiGetEvent (hdl, 0, &eData, &siEvent) == SI_IS_EVENT)
      {
      switch (siEvent.type)
        {
        case SI_MOTION_EVENT:
          break;

        case SI_ZERO_EVENT:
          break;

        case SI_BUTTON_EVENT:
          break;

        default:
          break;
        } /* end switch on Spaceball event type*/
      } /* end if the event was a Spaceball event */
    else
      {
      TranslateMessage (&msg);
      DispatchMessage (&msg);
      }
    }

  /*
   * Close connection to the Spaceball driver.
   */
  SiClose (hdl);
  SiTerminate ();
}

MFC

There are several ways to integrate the Spaceball into an MFC application. All of these methods are based on the Windows message sent by the SpaceWare driver to the application. The Windows message ID for these messages are allocated using the Windows call RegisterWindowsMessage. The results of this call are not portable between machines or even between Windows sessions. If you wish to find the Windows message ID for a particular session you can make the following call:

spwid = RegisterWindowMessage ("SpaceWareMessage00");

This will give you the message ID for the first Spaceball attached.

The first two approaches to integrating Spaceball support into an MFC application are based on overriding function calls within an MFC class. One approach is to override the PreTranslateMessage member of the CView class. This method is used in the sbtstmfc demo and in the following code sample.

The other approach is to override the WindowProc member of the Window(s), which should receive the Spaceball messages. This approach is similar to the first in that all the WindowProc member does is check to see if the given message is a Spaceball message and if not passes the message along to the base class WindowProc. The following code sample shows an example of this style of support:

/*
 * SpaceWare include file
 */

#include "si.h"

/*
 * appView class definition
 */

class appView: public CView
{ /* Application specific information */
// Overrides
//{{AFX_VIRTUAL(CsbtstmfcView)
  public:
  virtual BOOL PreTranslateMessage(MSG* pMsg);
  // To trap the Spaceball messages
  //}}AFX_VIRTUAL

SiHdl m_DevHdl; /* Handle to Spaceball Device */

 /* MFC Message map functions. */

protected:

  //{{AFX_MSG(appView)
  afx_msg int OnCreate(LPCREATESTRUCT lpCreateStruct);
  //}}AFX_MSG
  DECLARE_MESSAGE_MAP()

} /* end of appView class declaration */

 /*
  * Message map declaration in the appView.cpp file.
  */

BEGIN_MESSAGE_MAP(CsbtstmfcView, CView)
  //{{AFX_MSG_MAP(CsbtstmfcView)
  ON_WM_CREATE()
  //}}AFX_MSG_MAP
END_MESSAGE_MAP()

int appView::OnCreate( LPCREATESTRUCT lpCreateStruct )

{
  if (CView::OnCreate (lpCreateStruct) == -1)
    {
    return -1;
    }

  /*
   * Initialize Spaceball
   */

  SiOpenData oData; // OS Independent data to open Spaceball.
  SiInitialize (); // Initialize the SpaceWare input library.
  SiOpenWinInit (&oData, m_hWnd); // Initialize Windows specific data.
  if ( (m_DevHdl = SiOpen ("App name", SI_ANY_DEVICE, SI_NO_MASK,
                            SI_EVENT, &oData)) == NULL )

    {
    SiTerminate (); // Shut down the SpaceWare input library
    MessageBox ("Sorry - No supported Spacetec IMC device available.\n"
              NULL, MB_OK);

  if (m_hWnd != NULL)
    {
    DestroyWindow (); // Destroy window.
    }

  ExitProcess (1); // Exit program.
  }

/*
 * Turn the driver GUI on.
 */

  SiSetUiMode (&m_DevHdl, SI_UI_ALL_CONTROLS);
  return 0;

} /* end of OnCreate */

 
BOOL appView::PreTranslateMessage( MSG* pMsg )

{

  SiSpwEvent event; /* SpaceWare Event */
  SiGetEventData eData; /* SpaceWare Event Data */

  /*
   * Check if this is a Spaceball event
   */

  SiGetEventWinInit (&eData, pMsg->message, pMsg->wParam, pMsg->lParam);

  if (SiGetEvent (m_DevHdl, 0, &eData, &event) == SI_IS_EVENT)
    {
    switch (event.type)
      {
      case SI_MOTION_EVENT:
        break;

      case SI_ZERO_EVENT:
        break;

      case SI_BUTTON_EVENT:
       break;

      default:
        break;

      } /* end switch */

    return (TRUE);
    } /* end if the message was Spaceball event */

  else
    { /* Send the message on to the parent class.
    return CView::PreTranslateMessage(pMsg);
    }

  } /* end of PreTranslateMessage */
 

The second approach is to use MFC message maps. This method relies on the use of the ON_REGISTERED_MESSAGE message map macro. This macro allows an application to trap a specific registered Windows message and call a user defined message callback. The following code fragment shows an example using this approach.

/*
 * SpaceWare include file
 */

#include "si.h"

class appView: public CView

{ /* Application specific information */
// Overrides

SiHdl m_DevHdl; /* Handle to Spaceball Device */

/* MFC Message map functions. */
protected:
  //{{AFX_MSG(appView)
  afx_msg int OnCreate(LPCREATESTRUCT lpCreateStruct);
  afx_msg LONG OnSpaceball( WPARAM wParam, LPARAM lParam );
  //}}AFX_MSG

  DECLARE_MESSAGE_MAP()

} /* end of appView class definition */

/*
 * Message map declaration in the appView.cpp file.
 */

int WM_SPACEBALL = RegisterWindowMessage ("SpaceWareMessage00");

BEGIN_MESSAGE_MAP(CsbtstmfcView, CView)
  //{{AFX_MSG_MAP(CsbtstmfcView)
  ON_WM_CREATE()
  ON_REGISTERED_MESSAGE( WM_SPACEBALL, OnSpaceball )
  //}}AFX_MSG_MAP
END_MESSAGE_MAP()

int appView::OnCreate( LPCREATESTRUCT lpCreateStruct )
{
  if (CView::OnCreate (lpCreateStruct) == -1)
  {
  return -1;
  }

/*
 * Initialize Spaceball
 */

SiOpenData oData; // OS Independent data to open Spaceball.
SiInitialize (); // Initialize the SpaceWare input library.
SiOpenWinInit (&oData, m_hWnd); // Initialize Windows specific data.
if ( (m_DevHdl = SiOpen ("App name", SI_ANY_DEVICE, SI_NO_MASK,
                         SI_EVENT, &oData)) == NULL )
  {
  SiTerminate (); // Shut down the SpaceWare input library
  MessageBox ("Sorry - No supported Spacetec IMC device available.\n"
              NULL, MB_OK);
  if (m_hWnd != NULL)
    {
    DestroyWindow (); // Destroy window.
    }

  ExitProcess (1); // Exit program.
  }

/*
 * Turn the driver GUI on.
 */

  SiSetUiMode (&m_DevHdl, SI_UI_ALL_CONTROLS);
  return 0;

} /* end of OnCreate */

 
LONG appView::OnSpaceball( WPARAM wParam, LPARAM lParam )
{
  SiSpwEvent event; /* SpaceWare Event */
  SiGetEventData eData; /* SpaceWare Event Data */

  /*
   * Decode Spaceball event.
   */

  SiGetEventWinInit (&eData, WM_SPACEBALL, wParam, lParam);

  if (SiGetEvent (m_DevHdl, 0, &eData, &event) == SI_IS_EVENT)
    {
    switch (event.type)
      {
      case SI_MOTION_EVENT:
        break;

      case SI_ZERO_EVENT:
        break;

      case SI_BUTTON_EVENT:
        break;

      default:
        break;

      } /* end switch */

    } /* end if the message was Spaceball event */

  return (TRUE);

} /* end of OnSpaceball */

 

Building the Libraries

Before recompiling, ensure that the development environment is set properly. In particular, the paths must be set up to find the appropriate Microsoft libraries used by nmake. Normally this can be done by running the vcvars32.bat file in the msdev bin directory. Failing that, the environment variables need to be setup manually. For Visual C++ 4.2, the development environment is usually set up in the directory msdev on the main drive (usually C:\). Assuming this, for Windows NT 4.0, the following environment variables should be set in Control Panels > System under the Environment tab, in the User Variables section:

lib set to: C:\msdev\lib,C:\msdev\mfc\lib
include set to: C:\msdev\include,C:\msdev\mfc\include
add to path: C:\msdev\bin

Adjust these drives and directories to your own situation.

If you are using the Visual C++ 5.0 compiler, these variables are similar. The difference is that Visual C++ became part of Visual Studio and therefore the paths to get to the binaries and libraries are completely different. Go to the same control panel tab and set up the same environment variables as follows:

lib set to: C:\Program Files\DevStudio\Vc\lib,C:\Program Files\DevStudio\Vc\mfc\lib
include set to: C:\Program Files\DevStudio\Vc\include,C:\Program Files\DevStudio\Vc\mfc\include
add to path: C:\Program Files\DevStudio\Vc\bin

Note that under Windows 95, only the path variable in the autoexec.bat file needs to be changed. Using Visual C++ 4.2 add the following line:

PATH=%PATH%;C:\MSDEV\LIB;C:\MSDEV\MFC\LIB;C:\MSDEV\INCLUDE;C:\MSDEV\MFC\INCLUDE;C:\MSDEV\BIN

With Visual C++ 5.0 the following lines are required:

PATH=%PATH%;C:\PROGRA~1\DEVSTU~1\VC\BIN
PATH=%PATH%;C:\PROGRA~1\DEVSTU~1\VC\LIB;C:\PROGRA~1\DEVSTU~1\VC\MFC\LIB
PATH=%PATH%;C:\PROGRA~1\DEVSTU~1\VC\INCLUDE;C:\PROGRA~1\DEVSTU~1\VC\MFC\INCLUDE

To compile from the command line in Windows 95, you must set MS-DOS window environment memory to the maximum allowed value (4096).

To initiate a build, open a MS-DOS window and go to the directory that contains the extracted files. Start the build by typing build. This will rebuild the entire tree under either Visual C++ 4.2 or 5.0 (that which is present). As the build finishes it inserts the DLLs into the appropriate system directory. This copy operation is defined in the last few lines of the build.bat script. If the copy operation does not finish correctly, check the last few lines of build.bat and correct if necessary.

Alternatively you may rebuild each library by loading the makefile into the Microsoft Developer Studio (File > Open Workspace > List Files of Type > Makefiles).

When building an application, the SpaceWare libraries must be linked in. To do this in Visual C++ 4.2, select Build > Settings, choose the Link tab, and add one of the following files to the Object/Library module listing:

Single-threaded applications:
    %SDK%\lib\uapi\siapp\<platform>\sthread\release\siappst.lib
    %SDK%\lib\uapi\siapp\<platform>\sthread\debug\siappstd.lib

Multithreaded applications:
    %SDK%\lib\uapi\siapp\<platform>i\mthread\release\siappmt.lib
    %SDK%\lib\uapi\siapp\<platform>\mthread\debug\siappmtd.lib

The only difference under Visual C++ 5.0 is that you add these libraries under the Project > Settings menu.

Note: 
It is important to match single-threaded applications to the single-threaded static library and multithreaded applications to the multithreaded static library.
